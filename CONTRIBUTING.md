# Contributing

Thanks for working on turbalance Analytics. This guide keeps the repo easy to
audit and the history readable.

## Commit messages

Use [Conventional Commits](https://www.conventionalcommits.org/). Each commit
subject is `type(optional-scope): short imperative summary`.

Types:

- `feat` — a new capability
- `fix` — a bug fix
- `docs` — documentation only
- `test` — adding or fixing tests
- `chore` — tooling, deps, build, repo hygiene
- `refactor` — code change that neither fixes a bug nor adds a feature
- `perf` — performance improvement
- `ci` — CI/workflow changes

Examples:

```
feat(collector): add HMAC replay protection to the source-bundle gateway
fix(lakehouse): handle empty Parquet partitions in the retention job
docs: document the Redfish BMC evidence bridge
chore: stop tracking generated build/ artifacts
```

Write the body (after a blank line) to explain *why*, not *what*. Avoid generic
subjects like "update features" or "update documents" — they make history
impossible to audit.

## Pull requests

- Keep PRs focused on one logical change.
- The PR title follows the same Conventional Commits format as a commit subject.
- Run the suite before opening: `npm test` (or `node tests/run-all.js`).
- Run `make lint` to syntax-check the Node entry points.

## Generated artifacts

Do not commit anything under `build/` (it is generated by scripts and tests and
is gitignored). The same goes for `__pycache__/` and `*.pyc`. Never commit real
secrets, certificates, or `.env` files — use the `*.example` templates.

## Tests

JS suites run on Node's built-in `node:test`/`assert`. The lakehouse Python
services need their dependencies installed (`pip install -r
requirements-platform.txt`); the screenshot-QA step needs Playwright browsers
(`npx playwright install`).
